_renderQuery () String private

Inherited from WPRequest: lib/constructors/wp-request.js:343

Process the endpoint query's filter objects into a valid query string. Nested objects and Array properties are rendered with indexed array syntax.

Returns:

String:

A query string representing the specified filter parameters

Example:

_renderQuery({ p1: 'val1', p2: 'val2' });  // ?p1=val1&p2=val2
                            _renderQuery({ obj: { prop: 'val' } });    // ?obj[prop]=val

site
(
  • endpoint
  • routes
)
WP static

Defined in wp.js:82

Convenience method for making a new WP instance

Parameters:

  • endpoint String

    The URI for a WP-API endpoint

  • routes Object

    The "routes" object from the JSON object returned from the root API endpoint of a WP site, which should be a dictionary of route definition objects keyed by the route's regex pattern

Returns:

WP:

A new WP instance, bound to the provided endpoint

Example:

These are equivalent:

var wp = new WP({ endpoint: 'http://my.blog.url/wp-json' });
                            var wp = WP.site( 'http://my.blog.url/wp-json' );

WP.site can take an optional API root response JSON object to use when bootstrapping the client's endpoint handler methods: if no second parameter is provided, the client instance is assumed to be using the default API with no additional plugins and is initialized with handlers for only those default API routes.

These are equivalent:

// {...} means the JSON output of http://my.blog.url/wp-json
                            var wp = new WP({
                              endpoint: 'http://my.blog.url/wp-json',
                              json: {...}
                            });
                            var wp = WP.site( 'http://my.blog.url/wp-json', {...} );

validatePath () WPRequest chainable

Inherited from WPRequest: lib/constructors/wp-request.js:399

Validate whether the specified path parts are valid for this endpoint

"Path parts" are non-query-string URL segments, like "some" "path" in the URL mydomain.com/some/path?and=a&query=string&too. Because a well-formed path is necessary to execute a successful API request, we throw an error if the user has omitted a value (such as /some/[missing component]/url) or has provided a path part value that does not match the regular expression the API uses to goven that segment.

Returns:

WPRequest:

The WPRequest instance (for chaining), if no errors were found
